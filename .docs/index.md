# .docs 인덱스 (컨텍스트 라우터)

> 에이전트는 **필요한 문서만** 골라 읽는다. 모든 문서를 한꺼번에 컨텍스트에 올리지 않는다.
> 이 인덱스는 "지금 하려는 작업에 어떤 문서가 필요한가"를 빠르게 찾기 위한 지도다.

## 작업별 라우팅

| 지금 하려는 일 | 먼저 읽을 문서 |
|---|---|
| 아키텍처/설계 결정 이해 | `design/architecture.md`, `design/core-beliefs.md` |
| 기능/API 동작 확인 | `specs/` (예: `specs/api_specs.md`) |
| 진행 중 작업·기술부채 파악 | `plans/` (예: `plans/tech-debt.md`), 루트의 `PLAN.md` |
| 외부 라이브러리/디자인시스템 참조 | `references/` |

## 디렉터리 역할
- **design/** — 변하지 않는 설계 신념과 아키텍처 경계. "왜 이렇게 만들었나"의 단일 진실 공급원.
- **specs/** — 제품/기능/API 명세. "무엇이 어떻게 동작해야 하나".
- **plans/** — 실행 계획, 기술부채 트래커. 시간이 지나면 변하는 살아있는 문서.
- **references/** — 외부 자료(라이브러리 문서 발췌, `llms.txt` 등). 토큰 절약을 위해 발췌본 위주.

## 원칙 (하네스 엔지니어링)
- **계층화**: 거대 단일 파일을 만들지 않는다. 작업 맥락에 가까운 작은 문서로 쪼갠다.
- **선택적 로딩**: 관련 없는 문서를 미리 읽지 않는다. 토큰 = 비용.
- **단일 진실 공급원**: 같은 사실을 여러 문서에 중복 기재하지 않는다. 링크로 연결한다.
